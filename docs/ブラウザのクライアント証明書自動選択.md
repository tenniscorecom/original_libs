# ブラウザのクライアント証明書 自動選択（検討中）

作成日: 2026-07-14
背景: 社内システムの RPA（Edge）で、サイトが**クライアント証明書認証**を要求する。
PC には証明書が3種入っており、**作業内容によって使う証明書が違う**
（例: 検索は「検索用証明書」、ダウンロードは「ダウンロード用証明書」）。
今は人が証明書選択ダイアログで手動選択している。これを自動化できるかの検討。
関連: `comken/browser`、[プロジェクト規約](プロジェクト規約.md)

---

## 前提の整理（2種類の「証明書」を混同しない）

| | サーバー証明書（TLS 検証） | クライアント証明書（本件） |
|---|---|---|
| 何を証明する | 相手サーバーが本物か | **こちら（利用者）が誰か** |
| 関係する設定 | `IGNORE_CERTIFICATE_ERRORS` 等（監査 P2-004） | 証明書選択ダイアログ |
| 本書の対象 | ✗ | ⭕ |

本件は**クライアント証明書の選択**であり、サーバー TLS 検証（別の宿題）とは無関係。

## なぜ手作業になっているか

サイトが「クライアント証明書を出してください」と要求すると、Edge は
**証明書選択ダイアログ（Windows のネイティブ画面）**を出す。
これは HTML ではないため **Selenium から操作できない**（RPA が固まる典型）。
だから人が選んでいる。

---

## 解決策: `AutoSelectCertificateForUrls` ポリシー

Edge の企業ポリシーで「このURLではこの証明書を自動選択」を設定すると、
**ダイアログを出さずに自動選択**される。コード不要、レジストリに書くだけ。

- レジストリ: `HKCU\Software\Policies\Microsoft\Edge\AutoSelectCertificateForUrls`
  （**HKCU なら管理者権限不要**。HKLM は要管理者＝IT 依頼）
- 値名は `1`, `2`, `3`... で1件ずつ。中身は JSON 文字列:

```json
{"pattern":"https://社内システムA.example.com","filter":{"SUBJECT":{"CN":"証明書1のCN"}}}
```

- `filter` は証明書の **ISSUER（発行者）** / **SUBJECT（サブジェクト）** の
  `CN` / `O` / `OU` / `L` で絞れる。両方書くと AND 条件。
- **重要: フィルタに複数の証明書が一致すると、ダイアログが再び出る。**
  3種を確実に1枚へ絞るフィルタ（多くは SUBJECT の CN）にすること。

---

## ケース別の可否

### ケースA: 作業ごとに URL が違う（サイトA=証明書1、サイトB=証明書2…）

一番きれい。URL ごとに AutoSelect を1件ずつ登録すれば各サイトで自動選択。**設定一発で終わり。**

### ケースB: 同じ URL で作業によって証明書を変える（← 今回の状況）

AutoSelect は **URL で証明書を決める**ため、同じ URL に対して「検索はB / ダウンロードはC」を
**URL だけでは区別できない**。証明書を選ぶタイミングを**ブラウザ起動時**にずらして切り替える。

1. 検索ジョブの前に、レジストリの filter を**検索用証明書の CN** に書き換える
2. Edge 起動 → 検索 → Edge を閉じる
3. ダウンロードジョブの前に、filter を**ダウンロード用証明書の CN** に書き換える
4. Edge 起動 → ダウンロード → 閉じる

comken は `with EdgeDriver() as d:` でジョブごとに新しい Edge を起動するため相性がよい。
**「1ジョブ＝1証明書＝1回のブラウザ起動」**に揃える。

---

## 絶対に外せない制約

**ブラウザは一度選んだクライアント証明書を、そのセッション（起動〜終了）の間、
同じホストに使い続ける（キャッシュされる）。**

- ❌ 1回の Edge 起動の中で「検索(B)→ダウンロード(C)」と切り替えることはできない
      （同じサイトには最初に選んだ1枚しか出せない）
- ⭕ 検索(B)を1セッションで全部やって閉じる → 別セッションを開いてダウンロード(C)、
      と**セッションを分ければOK**（証明書ログインなら開き直しで自動再認証される）

→ 自動化の設計は「**検索フェーズ（証明書B）を終えて閉じる → ダウンロードフェーズ（証明書C）で開き直す**」の2段構え。

## レジストリ切り替えの注意（順次実行前提）

AutoSelect はユーザー全体で1つのポリシーなので、**同じ PC で2ジョブを同時実行すると競合する**。
ただし本運用は「8台が順番に1つずつ実行」で、1台あたり同時1ジョブ＝競合しない。今の運用ならこのまま使える。
（将来並列化するなら、ジョブ間の排他 or ダイアログ自動操作方式に切り替える）

## 最終手段: ネイティブダイアログの自動操作

ポリシーが使えない場合、証明書選択ダイアログを pywin32 / UI Automation で
「N番目を選んで OK」と操作する。comken の `WindowHandler`（win32gui）が土台に使えるが、
**タイミング依存で不安定**。ポリシー方式を強く推奨し、これは最後の手段。

## 先に確認すべきこと（実装方針の分岐点）

**その3枚は本当に「操作ごとに別の証明書」か？** 1枚に検索・ダウンロード両方の権限がある
だけのこともある。証明書の管理者に「権限が別々に分かれているのか、1枚に集約できないのか」を確認する。

- **別々の証明書が必須** → 本書の「ケースB：フェーズ分割＋起動前切り替え」で実装
- **1枚に集約できる** → AutoSelect を1件登録するだけで完了（切り替え不要・最も簡単）
