# 認証情報の公開鍵配布 — 設計メモ

作成日: 2026-07-13
背景: 会社の RPA 環境（WinActor onCloud → 実行PC 8台）へ ID・パスワード類を一括配布したい。
[Salesforce認証パターン.md](Salesforce認証パターン.md) の「補足: 8台への機密配布の選択肢」の**案B（公開鍵ハイブリッド）の詳細設計**。
実装先の想定: `comken.credentials` のサブモジュール（このリポジトリ）。

---

## 全体像

```
管理サーバー                          共有フォルダ                    実行PC × 8台
─────────────                ──────────────           ─────────────
機密を DPAPI ストアに登録      public_keys/                 [初回] setup:
        │                      ├ PC01.pem  ◀──────────  鍵ペア生成・公開鍵を提出
        │                      ├ PC02.pem                秘密鍵は DPAPI で保護して保存
        ▼                      └ ...
pack: 全公開鍵で暗号化 ──────▶ credentials.bundle  ────▶ [毎回] pull:
                                                          秘密鍵で復号 → ローカルの
                                                          DPAPI ストアへ取り込み
                                                          → 以降は今まで通り
                                                            Credentials("salesforce") で読む
```

**設計の肝は「配布 = ローカル DPAPI ストアの同期」と捉えること。**
実行PC側は bundle を復号したら既存の `save_credential()` でローカル登録する。
これで既存の `Credentials("salesforce")` API・`REQUIRED_CREDENTIALS` 宣言・
登録済みプロジェクトのコードが**一切変更なしで動く**。配布モジュールは「登録作業の自動化」に徹する。

---

## 技術選定

| 部品 | 選定 | 理由 |
|---|---|---|
| ハイブリッド暗号の共通鍵 | Fernet（`cryptography`） | AES + HMAC の認証付き暗号。改ざんは復号時に必ず検出される。API が簡単で誤用しにくい |
| 共通鍵の包み方 | RSA-OAEP（SHA-256）・鍵長 3072bit | `cryptography` 標準。X25519 より馴染みがあり説明しやすい |
| 秘密鍵の保管 | PEM を DPAPI（`win32crypt`）で暗号化 | 既存 `store.py` と同じ方式。鍵管理不要・マシン/ユーザー紐付け |
| 依存ライブラリ | `cryptography` + 既存の `pywin32` | **オフライン環境に cryptography の wheel を持ち込めるかが最初の関門**（無理なら案B自体を断念） |

## ファイル・フォーマット設計

### 共有フォルダのレイアウト

```
\\server\rpa-credentials\
  public_keys\
    PC01.pem            ← 各PCが setup で提出（COMPUTERNAME.pem）
    PC02.pem
  credentials.bundle    ← pack が生成する配布ファイル（JSON）
```

### bundle のフォーマット（JSON）

```json
{
  "version": 1,
  "created_at": "2026-07-13T10:00:00",
  "payload": "<Fernet で暗号化した機密 JSON を base64>",
  "wrapped_keys": {
    "PC01": "<Fernet キーを PC01 の公開鍵で RSA-OAEP 包みして base64>",
    "PC02": "..."
  }
}
```

- `payload` の平文は `{"キー名": "値", ...}`（`credentials.dat` の中身と同じ形式）
- 本文は1回だけ暗号化し、共通鍵だけを台数分包む（ハイブリッド暗号。ファイルサイズも処理も軽い）
- Fernet は認証付きなので payload の改ざんは復号時に例外になる。
  `wrapped_keys` を差し替えられても自分の秘密鍵で開かなくなるだけで、偽の機密を注入されることはない

### 実行PC側のローカルファイル

```
%USERPROFILE%\.comken\
  credentials.dat        ← 既存。取り込み先
  dist_private_key.dat   ← 新規。DPAPI で暗号化した秘密鍵 PEM
  dist_state.json        ← 新規。取り込み済み bundle の created_at（更新判定用。機密ではない）
```

## モジュール設計（comken への組み込み）

配置: `comken/credentials/dist.py` + CLI エントリ `python -m comken.credentials.dist`

### CLI は3コマンド

| コマンド | 実行場所 | 頻度 | やること |
|---|---|---|---|
| `setup` | 実行PC | 初回のみ | 鍵ペア生成 → 秘密鍵を DPAPI 保存 → 公開鍵を共有フォルダに提出 |
| `pack` | 管理サーバー | パスワード変更時 | ローカル DPAPI ストアから機密を読み、全公開鍵で bundle を生成 |
| `pull` | 実行PC | RPA 起動時（バッチ先頭） | bundle が更新されていれば復号してローカルへ取り込み |

### 公開関数（コードから呼ぶ場合）

```python
def setup_machine(shared_dir: str | Path) -> Path: ...
    """鍵ペアを生成し、公開鍵を shared_dir/public_keys に提出する。提出先パスを返す。"""

def pack_bundle(shared_dir: str | Path, prefixes: list[str]) -> Path: ...
    """ローカルの DPAPI ストアから prefixes に一致する機密を集めて bundle を書き出す。"""

def pull_bundle(shared_dir: str | Path) -> int: ...
    """bundle が前回より新しければ復号してローカル DPAPI ストアへ取り込む。取り込んだ件数を返す。"""
```

- `pack` の対象は**プレフィックスで宣言**する（`REQUIRED_CREDENTIALS` と同じ思想）。
  「管理サーバーのストア全部を無条件に配る」はやらない（配る必要のない機密まで8台に撒くのを防ぐ）
- 例外は `CredentialError` 配下に `DistributionError` を追加
  （MSG 定数に「何が・どこで・どうすればよいか」を書く規約に従う）
- `pull` は bundle の `created_at` を `dist_state.json` と比較し、未更新ならスキップ（毎回の起動を軽くする）

### 各処理の実装イメージ

**setup（実行PC・初回）**

```python
KEY_SIZE_BITS = 3072
PRIVATE_KEY_PATH = Path.home() / ".comken" / "dist_private_key.dat"

def setup_machine(shared_dir: str | Path) -> Path:
    private_key = rsa.generate_private_key(public_exponent=65537, key_size=KEY_SIZE_BITS)
    pem = private_key.private_bytes(  # 秘密鍵は PEM にして DPAPI で包む
        Encoding.PEM, PrivateFormat.PKCS8, NoEncryption()
    )
    encrypted = win32crypt.CryptProtectData(pem, "comken dist key", None, None, None, 0)
    PRIVATE_KEY_PATH.parent.mkdir(parents=True, exist_ok=True)
    PRIVATE_KEY_PATH.write_bytes(encrypted)

    public_pem = private_key.public_key().public_bytes(
        Encoding.PEM, PublicFormat.SubjectPublicKeyInfo
    )
    dest = Path(shared_dir) / "public_keys" / f"{os.environ['COMPUTERNAME']}.pem"
    dest.parent.mkdir(parents=True, exist_ok=True)
    dest.write_bytes(public_pem)  # 公開鍵は秘密ではないのでそのまま置いてよい
    return dest
```

**pack（管理サーバー）**

```python
def pack_bundle(shared_dir: str | Path, prefixes: list[str]) -> Path:
    shared_dir = Path(shared_dir)
    secrets = _collect_from_local_store(prefixes)  # 既存 store の _load_all() から絞り込み

    fernet_key = Fernet.generate_key()
    payload = Fernet(fernet_key).encrypt(json.dumps(secrets).encode("utf-8"))

    wrapped_keys = {}
    for pem_file in sorted((shared_dir / "public_keys").glob("*.pem")):
        public_key = serialization.load_pem_public_key(pem_file.read_bytes())
        wrapped = public_key.encrypt(
            fernet_key,
            padding.OAEP(mgf=padding.MGF1(hashes.SHA256()), algorithm=hashes.SHA256(), label=None),
        )
        wrapped_keys[pem_file.stem] = base64.b64encode(wrapped).decode("ascii")

    bundle = {"version": 1, "created_at": ..., "payload": ..., "wrapped_keys": wrapped_keys}
    # 書き込みは store.py と同じ「一時ファイル → os.replace」でアトミックに
    # （RPA 側が読んでいる最中に半端な bundle を見せない）
```

**pull（実行PC・起動時）**

```python
def pull_bundle(shared_dir: str | Path) -> int:
    bundle = json.loads((Path(shared_dir) / "credentials.bundle").read_text("utf-8"))
    if bundle["created_at"] <= _last_pulled_at():  # 未更新ならスキップ
        return 0

    my_name = os.environ["COMPUTERNAME"]
    if my_name not in bundle["wrapped_keys"]:
        raise DistributionError(DistributionError.MSG_NOT_REGISTERED.format(name=my_name))

    _, pem = win32crypt.CryptUnprotectData(PRIVATE_KEY_PATH.read_bytes(), None, None, None, 0)
    private_key = serialization.load_pem_private_key(pem, password=None)
    fernet_key = private_key.decrypt(base64.b64decode(bundle["wrapped_keys"][my_name]), OAEP(...))
    secrets = json.loads(Fernet(fernet_key).decrypt(base64.b64decode(bundle["payload"])))

    for name, value in secrets.items():
        save_credential(name, value)  # 既存 API で取り込み → 以降は今まで通り読める
    _save_pulled_at(bundle["created_at"])
    return len(secrets)
```

## セキュリティ設計の判断ポイント

1. **DPAPI はユーザースコープで使う（既存 store.py と同じ）。**
   → `setup` と `pull` と RPA 本体は**同じ Windows アカウントで実行する**こと。
   別アカウントで setup すると RPA から秘密鍵が復号できない。ここが一番ハマりやすい
2. **公開鍵すり替え対策が唯一の本質的な弱点。**
   攻撃者が `public_keys/EVIL.pem` を置けば次回の pack から機密を受け取れてしまう。対策:
   - `public_keys/` の書き込み権限を ACL で絞る（提出時だけ管理者が開ける運用でもよい）
   - pack 実行時に「今回包んだ PC 名一覧」をログに出し、台帳（8台のPC名）と目視照合する
   - より堅くするなら pack 側に許可 PC 名のホワイトリストを持たせる
3. **管理サーバーに平文が集まる構図は変わらない。** 配布方式をどう凝ってもここは同じ。
   管理サーバーのログイン制御・ストア（DPAPI）保護が本丸
4. **平文をログ・ファイルに出さない。** `pull` がログに書いてよいのはキー名と件数まで。
   値は logger に渡さない（既存規約どおり）
5. bundle・公開鍵は暗号文/非機密なので、共有フォルダ自体の秘匿性には依存しない
   （読まれても安全。書き込みだけ守る）

## 運用手順（できあがったときの姿）

| 場面 | やること |
|---|---|
| 新PC追加 | 新PCで `setup` → 管理者が公開鍵提出を確認 → 管理サーバーで `pack` し直し |
| パスワード変更 | 管理サーバーで該当キーを再登録 → `pack` → 各PCは次回起動時の `pull` で自動反映 |
| PC入替・再セットアップ | `setup` をやり直し（古い公開鍵は削除）→ `pack` し直し |
| PC撤去 | 公開鍵を削除 → `pack` し直し（以降の bundle をそのPCは開けなくなる） |
| 秘密鍵ローテーション | 「PC入替」と同じ手順。年1回など決めておく |

RPA 側は各シナリオのバッチ先頭に `python -m comken.credentials.dist pull` を1行足すだけ。
