# ライブラリ開発規約（comken 本体）

comken **本体（ライブラリそのもの）を編集する**ときの、ライブラリ固有のルールを定める。

> **言語レベルのコーディング規約（命名・型ヒント・定数・例外の書き方・ロギング・
> モジュール内の並び順など）は [CONVENTIONS.md](../CONVENTIONS.md) に従う。**
> 本書は「多くのプロジェクトから共有されるライブラリ」という立場ゆえに追加で守るルールだけを扱う。
> comken を"使う側"のプロジェクトを作るときは [プロジェクト規約.md](プロジェクト規約.md) を参照。

---

## 公開 API の管理

### `__all__` で公開範囲を明示する

```python
# comken/excel/__init__.py
from .handler import ExcelFile

__all__ = ["ExcelFile"]
```

`__all__` を書くと内部の実装クラスが外に漏れ出さない。**新しく公開したいものだけ**を `__all__` に足す。

### 内部ヘルパーには `_` プレフィックスを付ける

```python
class ExcelFile:
    def read_rows(self, sheet_name: str) -> list[tuple]:  # 公開メソッド
        ws = self._sheet(sheet_name)
        ...

    def _sheet(self, sheet_name: str):  # 外から直接呼ばないという意味
        ...
```

`_` 付きは「利用側が呼ぶことを想定しない」という宣言。将来シグネチャを変えても互換性を気にしなくてよい。

## カスタム例外の階層設計

すべての例外は `OriginalLibsError` を基底とする1本の階層にする。
利用側が `except OriginalLibsError` でライブラリ由来のエラーをまとめて捕捉できるようにするため。

```
OriginalLibsError                 ← すべての基底
├── ExcelError
│   ├── SheetNotFoundError
│   └── MacroError
├── CsvError
├── ColumnNotFoundError
├── ConfigError
├── SalesforceError
└── CredentialError
    ├── CredentialNotFoundError
    └── InvalidCredentialNameError
```

新しい例外を足すときのルール:

| ルール | 理由 |
|---|---|
| 必ず `OriginalLibsError`（または既存の中間クラス）を継承する | 利用側の `except OriginalLibsError` から漏れないようにする |
| 素の `Exception` / `ValueError` を投げない | 呼び出し側で「どのエラーか」を判別できるようにする |
| メッセージは `MSG_XXX` クラス定数のテンプレートにする | 文面を1か所で管理し、`.format(...)` で埋める（下記） |
| メッセージに「何が・どこで・どうすればよいか」を入れる | 非エンジニアがエラー画面を読んで対処できるようにする |

```python
class ExcelError(OriginalLibsError):
    MSG_FILE_NOT_FOUND = (
        "Excel ファイルが見つかりません: {path}\n"
        "パスが正しいか、ファイルが存在するかを確認してください。"
    )

# 送出側
raise ExcelError(ExcelError.MSG_FILE_NOT_FOUND.format(path=src))
```

- 例外は握りつぶさない（`except: pass` は禁止）。
- 行単位の処理エラーは行番号を含めて再送出する（`transfer_by_key` 等）。

## 破壊的変更と互換シム

comken は共有フォルダ配布で**全プロジェクトが常に最新を参照する**。
名前を黙って変えると、更新した瞬間に各プロジェクトが動かなくなる。だから旧名は消さずに残す。

| ルール | 内容 |
|---|---|
| 旧名には警告を付けて残す | `comken/deprecation.py` の `warn_renamed(旧名, 新名)` を使う |
| 警告は `FutureWarning` を使う | `DeprecationWarning` はデフォルト非表示で気づけない |
| 旧 import パスはシムにする | 下記のように `__getattr__` で新モジュールへ誘導する |

```python
# comken/salesforce/__init__.py（旧パスの互換シムの例）
from ..deprecation import warn_renamed
from ..salesforce_new import SalesforceApiClient as _SalesforceApiClient

def __getattr__(name):
    if name == "SalesforceApiClient":
        warn_renamed("comken.salesforce", "comken.salesforce_new")
        return _SalesforceApiClient
    raise AttributeError(name)
```

- 「利用者ゼロ」が確実な場合（リリース前・社内展開前）はシムを作らず単純に改名してよい。
- パッケージ名そのもの（`comken`）を変える場合は `rename_package.py` を使う（仕様書参照）。

## パッケージ構成

- **ライブラリのコードは `src/` レイアウトにしない。** リポジトリ直下に `comken/` を置く。
  各 PC が共有サーバーのリポジトリルートを `PYTHONPATH` で直接参照するため、`comken/` が
  直下にある構成を前提にしている（`rename_package.py`・pyproject も同様）。src レイアウトへの
  移行は「テスト隔離」という小さな利点に対し、参照フローを壊すリスクが大きく、YAGNI。
- 機能は「1フォルダ = 1関心事」で分ける（`csv/` `excel/` `windows/` `salesforce/` `browser/` 等）。
- 機能パッケージ同士は原則独立にする。依存は基盤（`exceptions` / `runtime` / `utils`）方向にだけ向ける。
  例外的な依存（`excel` → `windows` の COM フォールバック等）は遅延 import にして、
  片方が入っていない環境でも単体で動くようにする。
- 外部ライブラリ依存（`pypdf` `simple-salesforce` 等）を使うフォルダは、
  import 時に対処法つきのエラーを出し、そのフォルダを使わなければ影響しないようにする。

> `main.py` を `src/` に置かないのは**利用プロジェクト側**の話（[プロジェクト規約.md](プロジェクト規約.md)）。
> ライブラリ本体の「src レイアウトにしない」とは別のトピックなので混同しないこと。
