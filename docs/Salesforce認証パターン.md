# Salesforce 認証パターン整理

作成日: 2026-07-13（2026-07-15 更新）
背景: 会社の RPA 環境（WinActor onCloud → 実行PC 8台が順番に稼働）で Salesforce API を使う方針になった。
関連: [CONVENTIONS.md](../CONVENTIONS.md)、`comken/salesforce`

> **【決定・2026-07-15】** セキュリティトークンの使用が禁止（廃止）方向のため、
> **パターン2（OAuth 2.0 クライアントクレデンシャルフロー）を採用**。
> comken の `SalesforceApiClient` は client_id / client_secret + My Domain の URL で認証する
> 実装に変更済み。simple-salesforce ベースのクライアント（`SalesforceClient` /
> `SalesforceBulkClient`）は廃止・削除した。以下は選択の経緯として残す。
>
> **背景の裏付け:** Salesforce は旧認証（ユーザー名+パスワードの OAuth フロー・SOAP login）を
> 段階的に廃止しており、**2026-06-25 が一つの区切り**（既に経過）。移行先は
> クライアントクレデンシャル / JWT Bearer フロー。設定コンテナは
> **外部クライアントアプリ（External Client App）** が Connected App の後継として標準化中。
> 参考: note.com「2026年6月25日までに迫るSalesforce認証廃止への対応」
> （https://note.com/hiroshi_mzn/n/n0b3490b35bf7）

---

## 前提となる制約

- **DPAPI（comken.credentials）はマシン・ユーザー固有**。あるPCで暗号化した機密は別のPCでは復号できない。
  → 「管理サーバーで暗号化して8台に配布」は原理的に不可。成立するのは
  「管理サーバー側で復号し、WinActor がシナリオ変数として平文を実行PCに渡す」形。
- 実行PCはヘッドレス運用（人がブラウザでログイン操作をする前提は置けない）。
- **セキュリティトークン（パターン1）は使用禁止方向**のため、OAuth 2.0（パターン2）に移行する。

## パターン一覧

### パターン1: ユーザー名 + パスワード + セキュリティトークン（SOAP login）

simple-salesforce と同じ方式。`salesforce` が現在対応しているのはこれ。

| 項目 | 内容 |
|---|---|
| 保存が必要な機密 | ユーザー名・パスワード・セキュリティトークン |
| Connected App | 不要（Salesforce 管理者への依頼なしで始められる） |
| 初回セットアップ | セキュリティトークンをメールで受け取るだけ |
| 無人実行との相性 | ○（対話操作なし） |
| 失効リスク | **パスワード変更のたびにセキュリティトークンが再発行される**（両方の差し替えが必要） |
| 監査 | ログインしたユーザーとして記録される |

- 一番手軽だが、Salesforce はユーザー名+パスワード系の認証を段階的に縮小する方針。
  組織のセキュリティ設定（MFA強制・API制限）次第でブロックされる可能性がある。
- パスワード定期変更ポリシーがある組織だと、8台分の機密差し替え運用が定期的に発生する。

### パターン2: OAuth クライアントクレデンシャルフロー（推奨候補）

インテグレーションユーザー（RPA専用アカウント）を1つ用意し、Connected App の
client_id / client_secret だけでアクセストークンを毎回取得する。

| 項目 | 内容 |
|---|---|
| 保存が必要な機密 | client_id・client_secret のみ |
| Connected App | **必要**（管理者が「実行ユーザー」を指定して設定） |
| 初回セットアップ | Connected App 作成 + 実行ユーザー割り当て（管理者作業） |
| 無人実行との相性 | ◎（毎回トークンを取り直すだけ。長期保存トークンが存在しない） |
| 失効リスク | ほぼなし（client_secret のローテーションのみ） |
| 監査 | すべてインテグレーションユーザーの操作として記録される |

- **リフレッシュトークン問題が消える**のが最大の利点。保存するのは client_id/secret だけで、
  パスワード変更・トークン失効と無縁。
- 8台構成でも配る機密が2つで済み、全台共通でよい。
- `salesforce` への追加実装が必要（トークンエンドポイントへの POST 1本なので小さい）。

#### 仕組み

1. 実行時にトークンエンドポイントへ client_id / client_secret を POST する
2. アクセストークンと instance_url が返る（**リフレッシュトークンは発行されない**）
3. 以降の REST API 呼び出しに `Authorization: Bearer <アクセストークン>` を付ける
4. トークンが失効したら（401）、また 1 からやり直すだけ

トークンの有効期限は「実行ユーザーのプロファイルのセッションタイムアウト設定」に従う
（組織デフォルトは2時間程度）。失効しても再取得すればよいので、期限管理は不要。

#### Salesforce 側の設定手順（管理者に依頼する内容）

1. **インテグレーションユーザーを作成する**
   - RPA 専用アカウント。人間のアカウントを流用しない（退職・異動で死ぬ）
   - 「Salesforce Integration」ライセンス（API 専用・Enterprise 以上で無償枠あり）が使えるならライセンス費ゼロ
   - 権限セットで「API の有効化（API Enabled）」を付与し、
     必要なオブジェクトの参照/更新権限だけに絞る（最小権限）
   - 「API ログインのみのユーザー（API Only User）」にすると画面ログイン不可になり、
     パスワード漏えい時のリスクがさらに下がる
2. **接続アプリを作成する**
   > **【重要・新規はこちら】** Salesforce は接続アプリ（Connected App）の後継として
   > **外部クライアントアプリ（External Client App）** を標準化中。**新規に作るなら
   > External Client App を推奨**（旧 Connected App も当面は動くが、今後は非推奨方向）。
   > 用語は 1:1 で対応する（下記の Consumer Key/Secret・クライアントクレデンシャル
   > 有効化・Run As ユーザーは External Client App でも同じ）。
   - 「OAuth 設定の有効化」にチェック
   - コールバック URL は必須項目だが使わないのでダミーで可（例: `https://login.salesforce.com/services/oauth2/callback`）
   - OAuth 範囲（スコープ）に「API を使用してユーザーデータを管理する（api）」を追加
   - **「クライアントクレデンシャルフローを有効化」にチェック**
3. **実行ユーザーを割り当てる**
   - アプリの「ポリシーを編集」→「クライアントクレデンシャルフロー」の
     「別のユーザーとして実行（Run As）」に 1 で作ったインテグレーションユーザーを指定
   - ここを設定しないとフロー自体が動かない（エラー: `invalid_grant`）
4. **コンシューマ鍵・コンシューマの秘密を取得する**
   - アプリの設定画面から Consumer Key（= client_id）と
     Consumer Secret（= client_secret）をコピーして安全に受け渡す
   - 作成直後は反映に数分かかることがある

#### トークン取得リクエストの実際

```
POST https://<MyDomain>.my.salesforce.com/services/oauth2/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=<Consumer Key>
&client_secret=<Consumer Secret>
```

レスポンス（JSON）:

```json
{
  "access_token": "00D...",
  "instance_url": "https://<MyDomain>.my.salesforce.com",
  "token_type": "Bearer",
  "issued_at": "...",
  "signature": "..."
}
```

**注意: `login.salesforce.com` では動かない。** このフローは組織の **My Domain の URL 必須**。
Sandbox の場合は `https://<MyDomain>--<Sandbox名>.sandbox.my.salesforce.com`。
つまり事前に組織の My Domain 名を確認しておく必要がある。

Python での実装イメージ（requests のみ・comken スタイル）:

```python
import requests

TOKEN_URL = "https://<MyDomain>.my.salesforce.com/services/oauth2/token"
TIMEOUT_SECONDS = 30

def get_access_token(client_id: str, client_secret: str) -> tuple[str, str]:
    """アクセストークンと instance_url を返す。失効したら呼び直すだけでよい。"""
    res = requests.post(
        TOKEN_URL,
        data={
            "grant_type": "client_credentials",
            "client_id": client_id,
            "client_secret": client_secret,
        },
        timeout=TIMEOUT_SECONDS,
    )
    res.raise_for_status()
    body = res.json()
    return body["access_token"], body["instance_url"]
```

`salesforce` の `report.py` はすでに `instance_url + access_token` を受け取る設計なので
そのまま接続できる。`SimpleSF` / Bulk 側はログイン部分の差し替え
（SOAP login → 上記 POST）を追加すれば対応完了。実装コストは小さい。

#### 運用・セキュリティの注意

- **client_secret は実質パスワード**。これを持つ者はインテグレーションユーザーとして
  何でもできるので、扱いは従来のパスワードと同格にする（config.ini に書かない等）
- 8台への配布は「WinActor のシナリオ変数で渡す」か「各PCで DPAPI 登録」のどちらでも成立する。
  全台共通の2値だけなので、パターン1（PCごと・パスワード変更で差し替え）より運用がはるかに軽い
- 防御を足すなら: プロファイルの**ログイン IP アドレス制限**で実行PC群のIPだけ許可、
  Connected App 側の IP 制限緩和を「制限を適用」にする
- client_secret が漏えいした疑いがあれば Connected App からワンクリックで再生成
  （ローテーション）できる。差し替えは8台共通の2値のみ
- 実行ユーザーのセッションタイムアウトを極端に短くしすぎない
  （RPA の1バッチが完走できる長さは確保する）

#### 落とし穴チェックリスト

- [ ] My Domain の URL を使っているか（`login.salesforce.com` は不可）
- [ ] Connected App の「クライアントクレデンシャルフローを有効化」にチェックしたか
- [ ] 「別のユーザーとして実行（Run As）」を設定したか（未設定だと `invalid_grant`）
- [ ] 実行ユーザーに「API Enabled」があるか
- [ ] スコープに `api` を入れたか
- [ ] Connected App 作成直後は反映まで数分待つ

### パターン3: OAuth JWT ベアラーフロー

証明書（秘密鍵）で署名した JWT を送ってアクセストークンを得る。サーバー間連携の定石。

| 項目 | 内容 |
|---|---|
| 保存が必要な機密 | 秘密鍵ファイル |
| Connected App | 必要（証明書アップロード + プロファイル事前承認） |
| 初回セットアップ | 証明書の作成・登録・事前承認（管理者作業が最も多い） |
| 無人実行との相性 | ◎ |
| 失効リスク | 証明書の有効期限管理のみ |
| 監査 | JWT の sub に指定したユーザーとして記録（ユーザーを切り替え可能） |

- パターン2よりセキュリティ的に堅い（secret がネットワークを流れない）が、
  証明書と JWT 署名の実装・運用コストが上がる。
- 「ユーザーごとに成り代われる」ので、個人単位の監査が必要になった場合の逃げ道にもなる。

### パターン4: OAuth Web サーバーフロー + リフレッシュトークン

ブラウザで初回ログイン・同意し、リフレッシュトークンを保存して使い回す。

| 項目 | 内容 |
|---|---|
| 保存が必要な機密 | client_id・client_secret・リフレッシュトークン（ユーザーごと） |
| Connected App | 必要 |
| 初回セットアップ | **ユーザーごとにブラウザでの対話ログインが必要** |
| 無人実行との相性 | ×（初回・失効時に人手が必要） |
| 失効リスク | 管理者のセッション設定・明示的失効・パスワード変更で失効しうる |
| 監査 | 個人ユーザーとして記録される |

- 「個人個人でリフレッシュトークンを取得する」案はこれ。ヘッドレス8台と相性が最悪で、
  失効のたびに誰かがブラウザ操作する運用になる。**RPA 用途では非推奨**。
- 採用するのは「操作ログを個人に紐づける監査要件が絶対」の場合のみ。
  その場合もパターン3（JWT で sub を切り替え）の方が運用は楽。

#### よくある疑問: リフレッシュトークンで「なければならない」場合はあるか

基本的に**ない**。リフレッシュトークンは「対話ログインを毎回やらずに済ませるための道具」であって、
API の権能はどのフローで取ったアクセストークンでも変わらない。強いて必要になるのは:

1. **個人アカウントとして実行する監査要件がある** かつ **JWT フロー（パターン3）が許可されない**場合
   （JWT でも個人ユーザーに成り代われるので、通常は JWT で代替できる）
2. 組織のポリシーで Connected App の設定が Web サーバーフローしか許可されない場合
3. 利用したい機能がユーザーの対話セッション前提の場合（RPA の API 利用ではまず該当しない）

なお「リフレッシュトークンの方が安全」というのも誤解で、実体は**失効しにくい長期パスワード**。
保管の重みは client_secret と同格かそれ以上になる。

## 比較まとめ

| | 1. パスワード+トークン | 2. クライアントクレデンシャル | 3. JWT ベアラー | 4. リフレッシュトークン |
|---|---|---|---|---|
| Connected App | 不要 | 必要 | 必要 | 必要 |
| 無人実行 | ○ | ◎ | ◎ | × |
| 失効時の人手 | パスワード変更時 | ほぼなし | 証明書更新時 | 頻発しうる |
| 個人単位の監査 | ユーザー次第 | 不可（共通ユーザー） | 可（sub 切替） | 可 |
| comken 対応状況 | 対応済み | 未実装（小） | 未実装（中） | 未実装（大） |

## 現時点の推奨

1. **第一候補: パターン2（クライアントクレデンシャル + インテグレーションユーザー）**。
   リフレッシュトークンの保管・失効問題がそもそも発生しない。
2. Connected App を作れない（管理者に依頼できない）場合は、当面パターン1で開始し、
   組織のポリシーで塞がれた時点でパターン2へ移行する。
3. パターン4（個人リフレッシュトークン）は監査要件が確定するまで選ばない。

## 補足: 8台への機密配布の選択肢

Salesforce に限らず、ID・パスワード類を8台へどう配るかの選択肢。
前提: DPAPI はマシン固有なので、暗号化したものをそのまま配ることはできない。

### 案A: 各PCで手動登録（現行方式の横展開）

各PCで `python -m comken.credentials` を一度実行する。仕組みの追加ゼロで一番堅いが、
パスワード変更のたびに8台を回る。**配布物が少なく変更頻度が低いなら十分**。

### 案B: 公開鍵で暗号化して一括配布（ハイブリッド暗号）

各PCに鍵ペアを持たせ、管理サーバーが公開鍵で暗号化した機密ファイルを配る。
**詳細設計（ファイル形式・comken への組み込み・実装手順）は [認証情報の公開鍵配布.md](認証情報の公開鍵配布.md) を参照。**

1. **初回のみ**: 各PCでセットアップスクリプトを実行
   - 鍵ペアを生成し、秘密鍵は DPAPI で暗号化してローカル保存
   - 公開鍵（秘密ではない）を管理サーバーの共有フォルダに提出
2. **パスワード変更時**: 管理サーバーで1回入力するだけ
   - 配布スクリプトがランダムな共通鍵で機密一式を暗号化し、
     その共通鍵を8台分の公開鍵でそれぞれ暗号化して同梱（ハイブリッド暗号）
   - できた配布ファイルを共有フォルダ・NAS に置く（経路は暗号化済みなので安全）
3. **各PCの実行時**: 自分の秘密鍵で共通鍵を復号 → 機密を復号してメモリ上で使う

| 観点 | 評価 |
|---|---|
| 一括配布 | ◎ 管理サーバーで1回入力→全台に反映 |
| 初回セットアップ | 各PC1回だけ。公開鍵の受け渡しは平文でよいので楽 |
| 経路の安全性 | ◎ 共有フォルダに置くのは暗号文のみ |
| 実装コスト | `cryptography` ライブラリが必要（オフライン環境に wheel を持ち込めるか要確認） |
| 運用の穴 | PC再セットアップ時に鍵ペアの再生成・公開鍵の再提出が必要 |

注意点:
- **管理サーバーに平文が集まる構図は変わらない**。サーバーが破られたら全滅なのは
  どの案でも同じ（そこを守るのが本丸）
- 復号後の平文をファイルに書かない（メモリ上で使い捨てる）こと

### 案C: 全台共通のマスターキー（対称鍵）

各PCに同じマスターキーを初回だけ手動配布（DPAPI で保存）し、
管理サーバーはそのキーで暗号化したファイルを配る。案Bより実装が簡単だが、
初回のキー配布は安全な経路（手渡し等）が必要で、キーのローテーションは8台回ることになる。

### 案D: WinActor のシナリオ変数で実行時に渡す

管理サーバー側で復号し、WinActor が実行時に各PCへ変数として渡す。
PC側に機密が保存されないのが利点だが、平文がシナリオログ・変数ダンプに残らないかの確認が必須。

### 使い分けの目安

- **Salesforce をパターン2にするなら配布物は client_id/secret の2つだけ**で変更頻度も低い
  → 案A（各PC登録）で足りる可能性が高い。案Bはオーバーキル気味
- 他システムの ID/パスワードが多数あり、定期変更ポリシーで頻繁に差し替えるなら
  → 案B の価値が出る。comken に配布モジュールとして組み込む選択肢もある

      （なければインテグレーションユーザー1つで足りる）
